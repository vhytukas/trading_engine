## Why WASM
WebAssembly is used here to run the matching engine directly in the client as compiled Rust rather than proxying every order action through HTTP/WebSocket roundtrips. That design removes network latency from the critical interaction loop, reduces costly network serialization and transport overhead per UI event, and keeps state transitions deterministic by executing the same strongly typed engine logic in one local runtime boundary. Instead of chatty request-response flows, the UI performs in-process function calls into WASM and consumes compact snapshots, which significantly improves responsiveness while preserving correctness and reducing backend dependency for simulation, replay, and local strategy prototyping.
